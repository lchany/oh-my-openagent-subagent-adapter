VERL Subagent Union Workflow

Rollout8 运行前必确认项

本文档用于 Baseline 与 Optimized 单机或多机训练的强制入参确认。 在完整确认前，不得创建运行工作区、修改容器、写工作单、启动 Ray 或训练。

适用：单机 / 多机 模式：Baseline vs Optimized 确认协议：VERL_WORKFLOW_INTAKE_V1 确认令牌：CONFIRM_COMPLETE_INTAKE

一、不可跳过的确认规则

硬门禁：所有字段必须在同一份完整清单中给出具体值，并由用户精确回复 CONFIRM_COMPLETE_INTAKE。普通“确认”、部分字段确认或从历史记录推断，均不生效。

新运行原则

  • 每次运行必须有新的 run_id。
  • 每次运行必须使用新的绝对 workflow_workspace。
  • 不得静默复用旧 run、旧输出目录或旧 Ray session。
  • 已确认工作单不可改写；输入变化必须创建新 run。

交互边界

  • 所有必要确认只由主 Agent 与用户完成。
  • 子 Agent 禁止向用户提问，也禁止创建子 Agent。
  • 确认完成后，容器、Ray、重试和修复由工作流自主决策。
  • 任何超出授权范围的操作必须停止并返回主 Agent。

二、强制确认清单

1. 运行身份与工作区

  • 必填：run_id。
  • 必填：全新的绝对 workflow_workspace。
  • Baseline、Optimized、Controller、Scripts 四个工作区绝对路径。
  • 门禁脚本和容器创建脚本的绝对目标路径。
  • 确认目标目录不存在，避免覆盖未知文件。

2. 容器、源码与镜像

  • Baseline 和 Optimized 的容器名、节点映射及运行状态。
  • 每个容器的源码根，例如 /vllm-workspace/verl。
  • 镜像 ID、镜像归档路径和 Ascend runtime 方案。
  • 是否授权创建、替换或重启缺失/不兼容的角色容器。
  • 必须挂载并验证 /mnt/disk2t 与 /mnt/sfs_turbo。
  • Optimized 源码必须有 Git/patch 基准、关键哈希和实际运行导入证据。
  • Baseline 与 Optimized 必须使用独立容器和独立源码环境，避免共享源码污染。

3. 拓扑、节点和 NPU

  • 目标拓扑：单机、多机或其他明确拓扑。
  • 节点数量、每个节点的角色和用户确认的私网地址。
  • 每个节点的物理 NPU 编号，不能仅写“8卡”。
  • Baseline 与 Optimized 是顺序复用还是并行使用资源。
  • 禁止操作的卡组、容器和进程边界。
  • Ray Head/Worker、端口、临时目录和 session 的隔离策略。
  • 训练前必须证明目标卡空闲；清理前必须证明进程属于本 run。

4. 模型和数据集

  • 模型绝对路径。
  • 训练数据集绝对路径。
  • 验证数据集绝对路径，或明确填写 none。
  • 容器内必须能访问上述路径。
  • 不得从历史 run 推断或替换模型、数据集。

5. 完整训练参数

类别 必须确认的字段
训练规模 training_steps、epochs、从 step 1 启动、resume 策略
数据批量 train batch、validation batch、shuffle、filter、truncation
PPO 批量 ppo mini batch、ppo micro batch per GPU、dynamic batch 开关
Rollout rollout.n、rollout/ref logprob micro batch、backend、async 模式
序列限制 max prompt、max response、max model length、max batched tokens、max num seqs
并行与显存 TP、FSDP size、NPU per node、gpu memory utilization、offload/recompute
确定性 seed、模型路径、数据集路径、reward 函数
保存与恢复 resume=disable、save_freq=-1、checkpoint 和 optimizer 恢复策略

6. 指标与精度报告

  • 性能指标名称：单步耗时、训练吞吐或其他指标。
  • 每个指标的单位。
  • 明确聚合窗口，例如 steps 6–15。
  • Reward 指标名称及比较方向。
  • 精度策略：仅报告，或用户明确指定的判定策略。
  • 若为 report_only，只报告原值、绝对偏差和相对偏差，不替用户判断合理性。

7. 优化目标和允许差异

  • 明确本轮唯一优化目标，例如 Rollout8 Native-N v2。
  • 列出所有允许的 Baseline/Optimized 差异。
  • 除容器身份、输出路径和批准的优化字段外，训练字段必须一致。
  • 源码兼容修复若影响公共路径，必须说明是否两侧一致应用。
  • 比较前必须完成字段级审计和源码/运行时激活审计。

8. Launcher 与执行方式

  • 提供确切 launcher，或授权 Baseline Runner 生成最小确定性 launcher。
  • 训练只能通过工作区门禁脚本启动。
  • Optimized 必须继承 Baseline canonical launcher/config。
  • 不得继承旧 run 的 step、指标、checkpoint、optimizer、Ray session 或输出目录。

9. 重试与授权范围

  • 正整数 max_attempts。
  • 容器创建/重启、局部清理、源码修复、网络和 Ray 操作的授权范围。
  • 禁止操作的容器、节点、NPU 或外部任务。
  • Git 网络同步是否授权；未授权时只能使用本地 Git/patch。
  • 所有修复必须写入 issues.yaml，并由真实门禁重跑验证。

10. 逐 step 结果与归档策略

默认禁止创建逐 step checkpoint、snapshot、额外 result 文件或逐 step 归档。 不得把逐 step 数值写入 Project Memory、Experience Vault 或项目归档。
  • 允许保留训练框架原生日志。
  • 只生成最终聚合窗口结果和必要终态证据。
  • 只有用户对当前任务明确要求时，才允许额外保存逐 step 结果。

四、标准阶段顺序

Baseline Runner→ Supervisor→ Optimized Runner→ Supervisor→ Comparator→ Supervisor→ Reporter
每个 Runner 独立负责准备、门禁、启动、监控、最小修复和重试；Supervisor 只读审查终态证据。 前一阶段未完成清理证明时，后一阶段不得使用同一组 NPU。

六、启动前最终 Go / No-Go

  • 完整 intake 已获得精确确认令牌。
  • 工作单、门禁脚本和容器创建脚本哈希已验证。
  • 容器身份、镜像、源码根及两个共享挂载已验证。
  • 目标 NPU 空闲，禁止卡组和外部任务未被触碰。
  • Baseline/Optimized 字段差异仅包含批准项。
  • 没有继承历史 step、checkpoint、optimizer、Ray session 或输出目录。
  • 逐 step 结果不落盘、不归档；最终聚合策略明确。
  • 清理范围、重试次数和失败升级策略明确。
以上任一项缺失、矛盾或证据失效时，结论必须是 No-Go，不得启动训练。
